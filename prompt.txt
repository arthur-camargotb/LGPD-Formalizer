# Objetivo geral

Preciso criar um projeto em Python para transformar uma base SQLite real em uma base segura para demonstração comercial.

A base original será de um cliente real, mas será usada apenas internamente para demonstrar o aplicativo para um prospect da mesma área de negócio.

O prospect não receberá o aplicativo, não receberá a base de dados e não terá acesso técnico ao banco. Ele apenas verá as telas do aplicativo durante uma apresentação.

Mesmo assim, a base usada na demonstração não pode conter dados reais identificáveis, dados pessoais, dados sensíveis ou informações comerciais que possam revelar o cliente real.

O projeto deve entregar:

1. Código Python completo.
2. Estrutura de pastas recomendada.
3. Arquivos de configuração por entidade.
4. Tratamento de campos sensíveis.
5. Tratamento de tabelas que devem ser limpas.
6. Preservação ou regeneração controlada de chaves.
7. Propagação de chaves para tabelas dependentes, quando necessário.
8. Recalculo obrigatório de `rowkey`.
9. Logs seguros.
10. Validações técnicas.
11. `README.md` completo para manutenção futura.
12. Pasta `examples/` com arquivos de configuração de exemplo.

Não quero uma resposta conceitual, simplificada ou incompleta. Quero um projeto pronto para uso real, com código funcional, documentação e critérios claros de validação.

# Contexto LGPD

A solução deve seguir boas práticas técnicas compatíveis com a LGPD.

A ferramenta deve aplicar anonimização ou geração de dados sintéticos irreversíveis nos campos pessoais, identificáveis ou visíveis.

Importante:

1. Não deve ser possível reconstruir dados reais a partir da base final usando apenas a base sanitizada.
2. Não manter nomes reais.
3. Não manter e-mails reais.
4. Não manter telefones reais.
5. Não manter CPF real.
6. Não manter CNPJ real.
7. Não manter endereços reais.
8. Não manter observações livres com dados reais.
9. Não manter nomes de contatos reais.
10. Não manter nomes de representantes reais.
11. Não manter nomes de clientes reais.
12. Não manter informações comerciais específicas que possam identificar o cliente real.
13. Gerar dados fictícios válidos e coerentes para uso em tela e validações do software.
14. Gerar logs técnicos sem registrar dados reais sensíveis.
15. Explicar no `README.md` que a validação jurídica final depende do controlador, DPO ou jurídico da empresa.

A documentação deve deixar claro que preservar códigos técnicos pode ser adequado para manter a integridade do aplicativo, mas, se internamente esses códigos ainda permitirem associação com a base original, a base pode ser considerada pseudonimizada no ambiente interno, e não necessariamente anonimizada de forma absoluta.

Por isso, o objetivo prático da ferramenta é:

```text
Preservar a integridade técnica da aplicação, mas substituir todos os dados visíveis, pessoais, identificáveis ou comercialmente sensíveis por dados sintéticos coerentes.
```


# Política de chaves

As tabelas possuem chaves e relacionamentos entre si.

A regra padrão deve ser:

```text
Preservar chaves técnicas existentes.
```

Ou seja, por padrão, o script não deve alterar campos como:

```text
cdcliente
cdempresa
cdrepresentante
cdproduto
cdtabelapreco
cdpedido
cdcondicaopagamento
cdformapagamento
```

Motivo:

Alterar chaves aumenta muito o risco de quebrar integridade referencial entre tabelas e impedir o aplicativo de funcionar.

A anonimização deve focar principalmente nos dados visíveis, pessoais ou identificáveis.

---

# Configuração de política de chave

Cada entidade pode ter a seção:

```text
[key_policy]
mode=preserve
```

ou:

```text
[key_policy]
mode=regenerate
regenerate_fields=cdrepresentante
start_at=10
```

Modos suportados:

```text
preserve    -> mantém chaves originais e anonimiza apenas campos configurados
regenerate  -> gera novas chaves e propaga para tabelas dependentes
```

O modo padrão deve ser:

```text
preserve
```

A regeneração de chaves só deve ocorrer se configurada explicitamente.

---

# Exemplo de regeneração de chave

Arquivo `representante.txt`:

```text
[table]
name=tblvprepresentante

[primary_key]
fields=cdempresa,cdrepresentante

[key_policy]
mode=regenerate
regenerate_fields=cdrepresentante
start_at=10

[rowkey]
target=rowkey
fields=cdempresa,cdrepresentante
separator=;
trailing_separator=true

[foreign_keys]
cdempresa=tblvpempresa.cdempresa

[sensitive_fields]
nmrepresentante
dsemail
nufone
nucelular
nucpf
dsobservacao
```

Se a tabela original possui:

```text
cdempresa=1
cdrepresentante=847
```

E o novo código gerado for:

```text
cdempresa=1
cdrepresentante=10
```

O script deve criar um mapa interno:

```python
key_map = {
    "tblvprepresentante": {
        (1, 847): (1, 10)
    }
}
```

Esse mapa deve ser usado para atualizar todas as tabelas filhas configuradas.

Exemplo em `cliente.txt`:

```text
[foreign_keys]
cdempresa,cdrepresentante=tblvprepresentante.cdempresa,cdrepresentante
```

Então todo cliente que antes apontava para:

```text
cdempresa=1
cdrepresentante=847
```

deve passar a apontar para:

```text
cdempresa=1
cdrepresentante=10
```

Depois disso, qualquer `rowkey` afetada deve ser recalculada.

O script deve interpretar isso como:

```text

```

referenciando:

```text
tblvprepresentante.cdempresa + tblvprepresentante.cdrepresentante
```

O script deve validar se os campos existem tanto na tabela filha quanto na tabela pai.

Após o processamento, o script deve validar se as FKs configuradas continuam consistentes.

# Rowkey

A base possui uma coluna ou estrutura chamada `rowkey`, que representa a concatenação das chaves da tabela.

A `rowkey` precisa sempre ser atualizada/recalculada depois de qualquer alteração de chave.

Mesmo preservando chaves, o script pode recalcular a `rowkey` para garantir consistência.

Configuração padrão:

```text
[rowkey]
target=rowkey
fields=cdempresa,cdcliente
separator=;
trailing_separator=true
```

Exemplo:

```text
cdempresa=1
cdcliente=1234
```

Resultado:

```text
1;1234;
```

Sem separador final:

```text
[rowkey]
target=rowkey
fields=cdempresa,cdcliente
separator=;
trailing_separator=false
```

Resultado:

```text
1;1234
```

A `rowkey` pode ter mais campos:

```text
[rowkey]
target=rowkey
fields=cdempresa,nupedido,cdproduto,sequencia
separator=;
trailing_separator=true
```

Resultado:

```text
1;98765;10033;1;
```

O script deve:

1. Validar se a coluna configurada em `target` existe.
2. Validar se todos os campos em `fields` existem.
3. Recalcular a `rowkey` com base nos valores atuais dos campos.
4. Não usar a `rowkey` antiga como referência.
5. Respeitar `separator`.
6. Respeitar `trailing_separator`.
7. Registrar inconsistências no log.
8. Atualizar a `rowkey` depois de propagar chaves regeneradas.

# Caso a rowkey esteja em tabela separada

Se existir uma tabela separada chamada `rowkey`, e não uma coluna dentro das tabelas de negócio, o script deve suportar configuração específica.

Exemplo:

```text
[rowkey]
mode=external_table
table=rowkey
entity_field=nmtabela
entity_value=tblvpcliente
key_field=dsrowkey
fields=cdempresa,cdcliente
separator=;
trailing_separator=true
```

Nesse cenário, o script deve atualizar a tabela `rowkey` conforme a estrutura real configurada.

Se a estrutura da tabela `rowkey` não puder ser inferida com segurança, o script deve registrar aviso e exigir configuração explícita.

O script não deve apagar, recriar ou truncar a tabela `rowkey` sem autorização explícita.

# Respeito ao tipo do banco

O script não deve quebrar os tipos de dados.

Antes de atualizar um campo, deve identificar o tipo declarado da coluna via:

```sql
PRAGMA table_info(nome_tabela);
```

Se o campo for numérico, não gravar texto.

Se o campo for texto, pode gravar valores textuais.

Se o campo for `NOT NULL`, nunca gravar `NULL`.

Se o valor gerado exceder o tamanho aparente ou conhecido do campo, truncar com segurança, sem gerar valor inválido para documentos, e-mails ou dados que exigem formato.

# Consistência dos dados por registro

Exemplo:

```text
cdempresa=1
cdcliente=1234
```

Campos gerados:

```text
nmrazaosocial          -> Cliente Demo 001234
nmfantasia             -> Cliente Demo 001234
dsemail                -> cliente001234@demo.local
nucnpj                 -> CNPJ válido gerado de forma determinística
nufone                 -> telefone fictício coerente
rowkey                 -> 1;1234;
```

Preferencialmente, usar a chave primária configurada em `[primary_key]`.

Se não houver chave primária, usar `rowid`.

# Geração determinística

A geração deve ser determinística quando uma seed for informada.

Exemplo:

```bash
python main.py --seed 123
```

Para a mesma base, mesma configuração e mesma seed, o resultado deve ser o mesmo.

Usar uma combinação como:

```text
seed
nome_da_tabela
nome_do_campo
chave_do_registro
```

para gerar valores estáveis.

A solução deve funcionar com biblioteca padrão do Python.

Bibliotecas como `Faker` podem ser sugeridas apenas como opcionais, mas não devem ser obrigatórias.

# Validação de integridade referencial

Após processar a base, o script deve validar as relações configuradas em `[foreign_keys]`.

Exemplo:

```text
[foreign_keys]
cdempresa,cdrepresentante=tblvprepresentante.cdempresa,cdrepresentante
```

Validar se todo par:

```text
tblvpcliente.cdempresa + tblvpcliente.cdrepresentante
```

existe em:

```text
tblvprepresentante.cdempresa + tblvprepresentante.cdrepresentante
```

Se encontrar registros órfãos:

1. Registrar erro no relatório.
2. Em modo `--strict`, fazer rollback.
3. Em modo normal, registrar alerta crítico.

---

# Validação de rowkey

Após processar cada tabela com configuração de `rowkey`, validar se a coluna `rowkey` corresponde à concatenação dos campos atuais.

Exemplo:

```text
fields=cdempresa,cdcliente
separator=;
trailing_separator=true
```

Para:

```text
cdempresa=1
cdcliente=1234
```

Esperado:

```text
1;1234;
```

Se houver divergência, corrigir e registrar no log.
